┌─────────────────────────────────────────────────────────────────────┐
│                    Key Design Decisions                              │
└─────────────────────────────────────────────────────────────────────┘

1. Subprocess Integration
   ✓ Matches nextflow workflow patterns
   ✓ Ensures consistency with command-line usage
   ✓ Isolates script execution
   ✓ Easy to debug (separate processes)

2. PNG Heatmaps
   ✓ Faster loading than HDF5 in browser
   ✓ Standard format, works everywhere
   ✓ Can handle ~150MB files efficiently
   ✓ Browser's native image handling

3. On-Demand Processing
   ✓ Network graphs only when clicked
   ✓ Reduces initial load time
   ✓ Conserves memory
   ✓ Better user experience

4. Localhost-Only Default
   ✓ Security by default
   ✓ Prevents accidental exposure
   ✓ Configurable for advanced users
   ✓ Clear warning when network-accessible

5. Dual Cutoff Modes
   ✓ Euclidean: Simple, intuitive
   ✓ Coordinate: Fine-grained control
   ✓ Matches pair_features.py capabilities
   ✓ User can choose preference

File Processing:
  • Small mzML (<100MB):     ~1-2 minutes
  • Large mzML (>500MB):     ~5-10 minutes
  • Heatmap generation:      ~2-5 minutes
  • Feature extraction:      <1 minute
  • Feature pairing (2 files): ~1-2 minutes

GUI Responsiveness:
  • Heatmap load:            ~2-5 seconds
  • Feature overlay:         <1 second
  • Network graph (50 nodes): ~1-2 seconds
  • Diagnostic plot:         <1 second
  • Tab switching:           Instant

Memory Usage:
  • Base application:        ~100MB
  • Per heatmap loaded:      ~200-300MB
  • Per feature set:         ~50-100MB
  • Network graph cache:     ~10-50MB

Network Binding:
  Default:  127.0.0.1 (localhost only)
  Optional: 0.0.0.0 (all interfaces, with warning)

File Handling:
  • User uploads saved to dedicated directory
  • Processed files isolated in gui_data/
  • No system file access
  • Temp files cleaned up

Process Execution:
  • Scripts run with configured timeouts
  • stderr/stdout captured
  • Error handling for failures
  • No shell injection vulnerabilities

Data Validation:
  • File extension checks
  • Pickle safety (trusted sources)
  • Size limit awareness
  • Error recovery
